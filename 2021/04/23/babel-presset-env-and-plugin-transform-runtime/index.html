babel/preset-env与babel/plugin-transform-runtime差异

两者功能均是添加polyfill，差异是：
@babel/preset-env可以指定需要适配的浏览器版本，即为按需；
@babel/plugin-transform-runtime按core-js版本全量增加polyfill。

前言

一个生产上的项目被捕获到了babel-polyfill重复的异常，该项目使用了vue-cli@2系列构建的脚手架开发。

在历史vue脚手架中，bebel的配置大致如下：

分析

babel-preset-env与babel-plugin-transform-runtime均已停止维护。

babel-preset-env简介为：

A Babel preset that compiles ES2015+ down to ES5 by automatically determining the Babel plugins and polyfills you need based on your targeted browser or runtime environments.

babel-plugin-transform-runtime简介为：

Externalise references to helpers and builtins, automatically polyfilling your code without polluting globals. (This plugin is recommended in a library/tool)

两者功能均为自动添加polyfill，两者如果同时使用可能存在重复添加polyfill的情况。

vue-cli@3主要使用了@vue/babel-preset-app，其虽然也使用了@babel/plugin-transform-runtime，但其将配置为了corejs: false,不再会自动添加polyfill:

transform-runtime avoids inlining helpers in every file. This is enabled for helpers only, since polyfills are handled by babel-preset-env.

包体积明显增大。分析可知是因为fetch的缘故。@babel/preset-env对fetch添加了polyfill，而@babel/plugin-transform-runtime没有。似乎是因为fetch的缘故。core-js中提到了：

Missing polyfills⬆

  • ES String#normalize is not a very useful feature, but this polyfill will be very large. If you need it, you can use unorm.
  • ES Proxy can’t be polyfilled, you can try proxy-polyfill which provides a very little subset of features.
  • window.fetch is not a cross-platform feature, in some environments, it makes no sense. For this reason, I don’t think it should be in core-js. Looking at a large number of requests it might be added in the future. Now you can use, for example, this polyfill.
  • ECMA-402 Intl is missed because of the size. You can use those polyfills.

似乎是@babel/preset-env已经浏览器特性，对fetch使用到的promise进行了polyfill，然而Chrome > 58均已支持fetch和promise，这是为什么？难道是caniuse.com查询到的信息与babel查询到的信息不一致？

@babel/preset-env和@babel/plugin-transform-runtime都可以对promise添加polyfill，但是对fetch的处理存在差异。
